# 智能选股与策略自进化系统下一步落地详细计划

更新日期：2026-06-08

## 1. 当前阶段判断

当前系统已经具备智能选股与策略自进化的原型闭环：

- 自然语言要求可以通过规则解析或 DeepSeek 解析为 `StrategySpec`。
- 策略可以运行选股、保存候选、保存为策略。
- 策略可以启用、记录信号、回填 5 日/20 日前向收益。
- 策略健康度可以根据真实信号表现计算。
- 健康衰减时可以生成优化候选，并支持人工晋级或拒绝。
- 策略库页面已经能查看健康度、信号、优化记录、版本快照。

但系统距离“能长期稳定发现高质量机会、持续验证策略、自动优化策略、辅助盈利决策”仍有明显差距。下一阶段目标不是承诺盈利，而是把系统从“功能原型”推进到“可持续策略研究平台”。

## 2. 下一阶段总目标

下一阶段要完成一个可持续运行的研究闭环：

```text
数据更新
  -> 数据质量审计
  -> 因子计算
  -> 活跃策略选股
  -> 信号落库
  -> 前向收益回填
  -> 健康度评估
  -> 衰减识别
  -> 自动生成优化候选
  -> 回测验证
  -> 人工晋级/拒绝
  -> 版本沉淀
```

验收目标：

- 每日能自动或一键执行完整闭环。
- 每个策略都有可追溯的信号、收益、健康度、优化记录、版本记录。
- 每个选股结果和回测结果都有数据质量说明。
- 策略优化不再只看单次回测，而是结合样本外、真实信号表现和市场环境。
- 用户能清楚知道：为什么选出这些股票、策略是否有效、是否正在衰减、是否值得继续观察。

## 3. 优先级规划

### P0：补齐稳定闭环

P0 是下一步必须先做的内容。没有 P0，系统无法长期可靠运行。

#### 3.1 自动调度闭环

当前状态：

- 已有手动完整闭环 API。
- 还没有接入真正的定时任务。

要做：

- 在现有同步调度器后增加策略闭环任务。
- 每日数据更新完成后自动执行：
  - 回填信号前向收益。
  - 重新计算策略健康度。
  - 运行 active 策略生成当日信号。
  - 对 degraded/failing 策略生成优化候选。
- 增加任务运行记录表。
- 前端展示最近一次闭环执行结果。

建议新增表：

```sql
strategy_cycle_runs (
  run_id TEXT PRIMARY KEY,
  trigger_type TEXT,
  status TEXT,
  started_at TEXT,
  finished_at TEXT,
  total_strategies INTEGER,
  signal_runs INTEGER,
  health_checks INTEGER,
  optimization_jobs INTEGER,
  error TEXT,
  report_json TEXT
)
```

验收标准：

- 用户可以在页面点击“执行完整闭环”。
- 系统可以在每日同步后自动运行完整闭环。
- 每次闭环执行都有运行状态、耗时、错误、策略数量、生成信号数、优化任务数。
- 闭环失败不会影响下一次运行。

#### 3.2 数据质量审计接入策略结果

当前状态：

- 系统已有数据质量模块。
- 但智能选股和策略回测没有输出充分的数据质量解释。

要做：

- 每次策略选股结果增加数据质量摘要：
  - 股票池总数。
  - 有因子数据的股票数。
  - 因子缺失股票数。
  - A/B/C/D 数据质量分布。
  - 最新行情日期。
- 每次回测结果增加数据质量摘要：
  - 回测区间数据覆盖率。
  - 停牌/缺失交易日数量。
  - 是否存在异常价格。
  - 是否使用前复权数据。
- 在前端智能选股和策略库信号表展示质量提示。

验收标准：

- 用户能看到每次选股和回测的数据可信度。
- 数据质量差时，策略评级和健康度要降低置信度。
- 数据缺失严重时，系统明确提示“不建议依据该结果决策”。

#### 3.3 健康度评分重构

当前状态：

- 健康度已基于真实信号收益。
- 但评分规则还比较粗糙。

要做：

- 健康度拆成多个子分：
  - 信号活跃度。
  - 信号成熟度。
  - 胜率。
  - 平均收益。
  - 最大回撤。
  - 连续亏损。
  - 数据质量。
  - 市场环境适配度。
- 健康度输出结构化解释。
- 健康度不再只输出一个字符串列表。

建议模型：

```json
{
  "health_score": 76.5,
  "status": "healthy",
  "sub_scores": {
    "signal_activity": 80,
    "signal_maturity": 70,
    "win_rate": 75,
    "return": 78,
    "drawdown": 82,
    "data_quality": 90,
    "market_fit": 65
  },
  "degradation_reasons": [],
  "recommended_actions": []
}
```

验收标准：

- 前端可以展示健康度拆解。
- 用户能知道策略为什么健康或为什么衰减。
- 优化触发不再只看总分，而是看具体衰减原因。

## 4. P1：增强策略研发能力

P1 目标是让系统从“执行用户策略”升级为“辅助生成和改进策略”。

### 4.1 DeepSeek 策略生成增强

当前状态：

- DeepSeek 已接入自然语言解析。
- 目前主要是把用户要求转成 `StrategySpec`。

要做：

- 增加“策略生成模式”：
  - 用户输入目标，例如“中线低回撤策略”。
  - DeepSeek 生成 3-5 个候选策略。
  - 每个策略包含适用市场、核心假设、入场条件、退出条件、风险条件。
- 增加策略生成约束：
  - 只能使用系统已有因子。
  - 不能虚构数据源。
  - 必须输出可回测 `StrategySpec`。
- 增加“澄清问题”机制：
  - 周期不明确时询问。
  - 风险偏好不明确时询问。
  - 股票池不明确时询问。

新增 API 建议：

```text
POST /api/v1/intents/generate-strategies
POST /api/v1/intents/clarify
POST /api/v1/intents/validate-spec
```

验收标准：

- 用户输入一个策略目标，系统返回多个可执行策略。
- 每个策略都能直接运行选股和回测。
- DeepSeek 输出非法字段时会被拒绝或自动修正。

### 4.2 策略批量回测与筛选

当前状态：

- 单策略可以回测。
- 优化器可以搜索参数。

要做：

- 对多个候选策略批量回测。
- 输出策略排行榜。
- 支持按以下指标排序：
  - 样本外年化收益。
  - 最大回撤。
  - 夏普。
  - 胜率。
  - 信号数量。
  - 数据质量。
  - 稳定性得分。
- 自动过滤明显过拟合策略。

验收标准：

- DeepSeek 生成的多个策略可以一键批量回测。
- 系统自动推荐最值得观察的 1-2 个策略。
- 不合格策略不会进入策略库。

### 4.3 策略版本对比增强

当前状态：

- 已有版本快照。
- 前端只展示基础差异。

要做：

- 增加版本对比接口。
- 对比内容包括：
  - 入场条件差异。
  - 排序规则差异。
  - 仓位规则差异。
  - 退出规则差异。
  - 回测指标差异。
  - 实盘信号表现差异。
- 增加回滚版本能力。

建议 API：

```text
GET /api/v1/strategy-library/{strategy_id}/versions/{version_id}/compare
POST /api/v1/strategy-library/{strategy_id}/versions/{version_id}/rollback
```

验收标准：

- 用户能清楚看到优化候选改了什么。
- 用户能回滚到旧版本。
- 晋级和回滚都有审计记录。

## 5. P2：增强预测与盈利研究能力

P2 是更接近最终目标的能力，但必须建立在 P0/P1 稳定后。

### 5.1 单股预测模型

当前状态：

- 单股分析有趋势、置信度、支撑压力等。
- 但不是严格预测模型。

要做：

- 建立监督学习数据集：
  - 特征：技术因子、波动、趋势、成交量、市场环境、行业强弱、数据质量。
  - 标签：未来 5/10/20 日收益、是否跑赢基准、最大回撤。
- 训练基础模型：
  - Logistic/LightGBM/XGBoost 可选。
  - 先做离线训练，不接入实盘。
- 输出：
  - 上涨概率。
  - 跑赢基准概率。
  - 下跌风险概率。
  - 置信度。
  - 主要贡献因子。

验收标准：

- 每个预测结果都有历史验证指标。
- 模型输出不是拍脑袋，而是有训练样本和回测报告。
- 页面明确展示模型版本和适用范围。

### 5.2 市场环境与策略适配

当前状态：

- 已有市场环境页面。
- 策略健康度没有深度结合市场环境。

要做：

- 给每个策略打环境标签：
  - 趋势市。
  - 震荡市。
  - 高波动。
  - 低波动。
  - 成长风格。
  - 价值风格。
- 每次策略信号记录当日市场环境。
- 健康度分析区分：
  - 策略自身衰减。
  - 市场环境不适配。
- 优化时避免把市场环境切换误判为策略失效。

验收标准：

- 策略库可以显示策略适合什么市场。
- 策略失效原因可以区分“策略变差”和“市场不适配”。

### 5.3 组合级风控

当前状态：

- 策略主要按单策略、单候选维度工作。

要做：

- 增加组合维度：
  - 多策略信号合并。
  - 行业集中度。
  - 单股集中度。
  - 风格暴露。
  - 最大仓位。
  - 最大回撤预算。
- 输出组合风险提示。

验收标准：

- 系统不只是选股票，还能判断组合是否过于集中。
- 多策略同时发出同一股票信号时能合并评分。

## 6. P3：工程化与稳定性

### 6.1 正式测试体系

当前状态：

- 有 smoke 脚本。
- 缺正式测试套件。

要做：

- 引入 pytest。
- 增加测试：
  - DeepSeek 解析兜底测试。
  - StrategySpec 校验测试。
  - 策略信号落库测试。
  - 健康度历史测试。
  - 优化晋级/拒绝测试。
  - 回测成本和涨跌停测试。
  - API 导入测试。
- 前端增加关键页面构建测试。

验收标准：

- 每次改动可运行一条命令验证核心链路。
- 数据库表变更有迁移 smoke。

### 6.2 任务和错误监控

要做：

- 统一记录后台任务。
- 统一记录错误。
- 前端展示最近任务失败原因。
- 增加重试能力。

验收标准：

- 用户能知道系统昨天是否成功更新数据和运行策略。
- 失败时能看到失败在哪一步。

## 7. 推荐执行顺序

建议按以下顺序推进：

1. 自动调度闭环。
2. 数据质量接入选股和回测。
3. 健康度评分重构。
4. DeepSeek 多策略生成。
5. 多策略批量回测和排行榜。
6. 策略版本对比和回滚。
7. 单股预测模型。
8. 市场环境适配。
9. 组合级风控。
10. 正式测试体系和任务监控。

## 8. 下一次迭代建议

下一次建议先实现 P0 的三项：

- 自动调度闭环。
- 数据质量接入策略结果。
- 健康度评分重构。

这三项完成后，系统会从“手动研究工具”进一步变成“每天可自动运行、可追踪、可解释”的策略研究平台。
