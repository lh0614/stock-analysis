# 本地股票分析项目差距与盈利闭环思考

分析日期：2026-05-27

## 1. 项目当前基线

该项目已经不是纯原型，核心形态是“本地优先的股票分析工作台”：

- 前端：Vue 3 + Element Plus，已有分析驾驶舱、选股器、策略工坊、预警、数据同步、自选股、运行历史等页面。
- 后端：FastAPI，接口覆盖行情、指标、分析流水线、策略、选股、新闻、预警、同步调度。
- 数据：东方财富、AKShare、Baostock 三类公开数据源；日 K 线缓存为本地 pkl 文件；SQLite 负责工作流、策略、运行记录、股票池、预警等元数据。
- 分析：MA / MACD / RSI、支撑压力、短中长方向卡、简单波动区间预测、内置/上传策略脚本沙箱执行。
- 自动化：已有股票池同步、全量/增量 K 线同步、SSE 进度、断点续跑、每日定时同步。

需要澄清一点：代码里已经有 SQLite，因此“当前没有数据库”更准确地说是“没有专业行情/因子/财务数据库”。现有 SQLite + pkl 能支撑个人本地 MVP，但不足以支撑可靠投研、批量回测、数据血缘、复权一致性和长期盈利验证。

## 2. 顶级股票分析工具的能力参照

这里不按“是否能交易”对标，而按“分析能力”对标：

- Bloomberg Terminal：核心优势是综合数据、新闻、研究、分析与专业社区的统一终端，偏机构级全资产分析。
- Wind 金融终端：强在中国本土市场数据、7x24 新闻、研报、组合监控、风险指标和机构内容生态。
- FactSet Workstation：强在全球数据、公司/行业/组合分析、风险归因、筛选、模型和机构工作流。
- TradingView：强在交互式图表、海量指标/策略、Pine Script、云端预警、多周期图表、回放和社区。

## 3. 差距清单

### 3.1 数据层差距

| 维度 | 当前项目 | 顶级工具 | 差距 |
|---|---|---|---|
| 行情粒度 | 主要是日 K | 日内、分钟、tick、盘口、逐笔、Level 2、跨市场 | 无法做短线微观结构、盘中信号、滑点评估 |
| 数据稳定性 | 公开接口 + failover | 商业授权、专线/低延迟、多供应商校验 | 公开源可能限流、字段变化、停更、延迟不透明 |
| 数据落地 | pkl + SQLite 元数据 | 时序库、列式库、因子库、对象存储、血缘系统 | 查询、批量扫描、回测和可追溯性不足 |
| 复权与公司行动 | 默认前复权，缺少统一公司行动库 | 拆分、分红、配股、停复牌、指数调整、成分变更 | 回测易出现未来函数和价格不连续 |
| 基本面 | 基本缺失 | 财报、估值、盈利预测、一致预期、行业对比 | 长线价值分析不足 |
| 资金流 | 基本缺失 | 主力资金、北向、融资融券、龙虎榜、ETF 申赎等 | A 股情绪和资金面判断弱 |
| 新闻事件 | 简单新闻/公告时间线 | 7x24 新闻、研报、事件日历、舆情、情感分析 | 事件驱动能力弱，无法系统识别风险 |
| 数据质量 | 仅最小 bars 校验 | 完整性、异常值、复权一致性、跨源对账、延迟监控 | “数据准不准”没有可量化证明 |

### 3.2 分析与策略差距

| 维度 | 当前项目 | 顶级工具/成熟投研系统 | 差距 |
|---|---|---|---|
| 指标库 | MA、MACD、RSI | 数百指标、形态识别、量价结构、市场宽度、风格因子 | 技术分析覆盖面偏窄 |
| 多周期分析 | 短/中/长方向卡 | 多级别联动、日内到月线共振、板块/指数环境联动 | 当前方向较孤立 |
| 预测能力 | 历史波动区间 | 概率模型、情景分析、因子模型、机器学习、误差评估 | 预测没有训练、验证和置信校准 |
| 选股器 | 少量预设交集 | 因子筛选、排名、分组回测、行业中性、组合构建 | 只能“找形态”，不能证明有效 |
| 回测 | 基本缺失 | 事件驱动回测、交易成本、滑点、停牌涨跌停处理、组合回测 | 无法判断策略是否有正期望 |
| 策略平台 | 可上传 Python/ZIP，简单沙箱 | 策略版本、参数寻优、A/B、走样检测、归因 | 缺少策略生命周期管理 |
| 风控 | 预警规则较简单 | 仓位、回撤、波动、相关性、行业暴露、止损/止盈纪律 | 缺少“亏损控制系统” |
| 复盘 | 运行历史 | 交易日志、决策依据、盈亏归因、错误标签 | 无法形成学习闭环 |

### 3.3 产品体验差距

| 维度 | 当前项目 | 顶级体验 | 差距 |
|---|---|---|---|
| 图表 | 有 StockChart，但整体仍偏基础 | 多图联动、画线、指标叠加、回放、模板、快捷键 | 图表工作台不够强 |
| 响应速度 | 本地缓存后可用 | 秒开、增量刷新、后台预计算、懒加载 | 大规模扫描和首次同步仍重 |
| 可解释性 | 有方向卡和依据链 | 每个结论可追溯到数据、公式、版本、历史准确率 | 解释还没有形成审计链 |
| 工作流 | 有模板和断点 | 用户画像、默认模板、跨标的对比、任务编排 | 仍是单标的分析为主 |
| 报告 | 暂缺系统报告 | 一键日报、周报、组合报告、研报式导出 | 分析结果难沉淀 |

### 3.4 不能接券商柜台带来的差距

不能接券商柜台并不是致命问题，但会带来边界：

- 不能拿真实成交、委托、持仓和资金流水，无法自动计算真实滑点、成交偏差、持仓风险。
- 不能自动交易，盈利闭环必须靠“信号 -> 人工执行 -> 手工/导入复盘”完成。
- 不能做实盘风控拦截，只能做事前预警、模拟组合和复盘提醒。
- 不能验证盘中成交可得性，策略回测必须保守加入交易成本、涨跌停、停牌、成交额容量约束。

因此产品定位应明确：不是交易终端，而是“选股、分析、预警、计划、模拟、复盘”的决策辅助系统。

## 4. 如何做好股票分析去盈利

盈利不是来自“预测明天涨跌”，而来自一套可重复的正期望流程：

1. 先定义交易/投资风格

不要让一个系统同时随意服务短线、波段、长线。每类风格的胜率、赔率、持仓周期、数据需求完全不同。建议先固定三套独立工作流：

- 短线：市场环境、板块强度、量价突破、换手、资金流、涨跌停约束、次日风险。
- 中线：趋势、回撤位置、成交量结构、行业景气、资金持续性、止损位。
- 长线：商业质量、财务趋势、估值、行业空间、盈利预测、分红与现金流。

2. 建立“市场环境过滤器”

很多策略亏损不是个股错，而是市场状态错。每次个股分析前，应先判断：

- 指数趋势：上证、深成指、创业板、科创、沪深 300、中证 500/1000。
- 市场宽度：上涨家数、涨停/跌停、20 日新高/新低、均线多头比例。
- 风格：大盘/小盘、成长/价值、周期/消费/科技。
- 板块强度：行业涨跌、资金流、相对强度。

只有市场环境允许时，才放大个股信号；环境差时降低仓位或只观察。

3. 用“赔率 + 胜率 + 风险”替代“看多/看空”

每个候选股票都应输出：

- 入场触发条件：不是当前就买，而是什么条件成立才行动。
- 失效条件：跌破哪个位置、放量失败、消息证伪、指数破位等。
- 目标区间：第一阻力、第二阻力、估值上沿。
- 预期收益/预期亏损：至少要求收益风险比大于 2:1。
- 胜率来源：历史同类形态、因子分层、回测统计，而不是主观置信度。

4. 每个策略必须经过样本外验证

最低要求：

- 样本内开发：用于提出规则。
- 样本外验证：用于检查是否过拟合。
- 滚动回测：按年份/市场阶段检查稳定性。
- 分组统计：按行业、市值、波动率、牛熊市拆分。
- 加入成本：佣金、印花税、滑点、涨跌停无法成交、停牌。

没有回测和复盘的策略，只能叫“观点”，不能叫“系统”。

5. 盈利闭环应做成产品流程

建议闭环：

```text
数据同步 -> 数据质检 -> 市场环境 -> 股票池过滤 -> 因子/规则打分
-> 候选列表 -> 单票深度分析 -> 交易计划 -> 预警触发
-> 人工执行/模拟记录 -> 复盘归因 -> 策略修正
```

其中“交易计划”和“复盘归因”是当前项目最缺的盈利环节。即使不能接券商，也可以让用户手工记录：

- 计划入场价、止损价、目标价、仓位比例、理由。
- 实际入场/出场价、时间、盈亏、是否按计划执行。
- 错误标签：追高、止损慢、信号失效、市场环境差、数据误判、消息冲击。

长期盈利来自减少重复错误，而不是提高 UI 上的“置信度”。

## 5. 如何获取准确数据

在不能买专业数据库、不能接券商柜台的前提下，目标不是“绝对准确”，而是建立“可校验、可回滚、可标注置信度”的数据系统。

### 5.1 数据源策略

建议按数据类型分层：

| 数据 | 可用来源 | 建议 |
|---|---|---|
| 日 K | 东方财富、AKShare、Baostock、交易所公开数据 | 三源对账，保存原始值和标准化值 |
| 股票列表 | AKShare、交易所列表、东方财富 | 每日同步，记录新增/退市/改名 |
| 指数与行业 | 中证/申万/东方财富/AKShare | 固定行业分类版本，避免历史漂移 |
| 财务报表 | 巨潮资讯、交易所、AKShare、Tushare 可选 | 优先原始公告，第三方作补充 |
| 公告 | 巨潮资讯、上交所、深交所、北交所 | 结构化标题、日期、类型、附件链接 |
| 新闻 | 东方财富、财联社等公开源 | 标注来源和发布时间，不直接当事实 |
| 资金流 | 东方财富/同花顺公开口径 | 不同平台口径差异大，只作辅助 |

如果以后允许付费但仍不接券商，优先考虑数据服务，而不是交易柜台：Tushare Pro、聚宽数据、米筐、掘金、Wind/Choice/iFinD 导出数据等。

### 5.2 本地数据架构建议

当前 pkl + SQLite 可继续保留，但建议升级为：

- SQLite：保留配置、任务、策略、运行记录、用户输入、复盘记录。
- DuckDB：做本地分析查询、因子计算、批量选股、回测结果聚合。
- Parquet：存 K 线、财务、因子、新闻事件等列式数据。
- 原始数据区 raw：每次接口返回先原样落盘，便于回查。
- 标准化数据区 curated：清洗、复权、字段统一后的数据。
- 质量报告区 quality：缺失、异常、跨源差异、最新交易日延迟。

建议目录：

```text
data/
  raw/
    eastmoney/klines/
    akshare/klines/
    baostock/klines/
  curated/
    daily_bars.parquet
    stock_universe.parquet
    fundamentals.parquet
    events.parquet
    factors.parquet
  quality/
    daily_quality_report.parquet
  lsa.db
```

### 5.3 数据准确性规则

必须增加数据质量规则：

- 完整性：交易日是否缺 bar，上市前后是否异常。
- 唯一性：同一 symbol + trade_date 只能有一条标准记录。
- 合理性：high >= max(open, close)，low <= min(open, close)，volume >= 0。
- 连续性：停牌允许缺失，但要有停牌标记。
- 新鲜度：最新交易日是否已更新，区分盘中/盘后/节假日。
- 跨源一致性：收盘价、成交量、成交额差异超过阈值则标红。
- 复权一致性：qfq/hfq/raw 独立保存，不混用。
- 公司行动：分红、送转、配股、退市、改名必须记录生效日。

每次分析输出都应带上数据质量等级：

```text
A：三源一致，最新交易日完整
B：两源一致或轻微缺失，不影响分析
C：单源数据或存在异常，只能参考
D：数据缺失/冲突严重，禁止给方向结论
```

## 6. 优先建设路线

### 第一阶段：把“数据准”和“可回测”补起来

优先级最高，因为没有它，后续所有分析都可能是错的。

- 引入 DuckDB + Parquet，保留 SQLite 元数据。
- 建 daily_bars 标准表：symbol、trade_date、open、high、low、close、volume、amount、adjust、source、quality_score。
- 每日同步后生成质量报告。
- 做跨源对账和异常标记。
- 建基础回测引擎：单票/组合、成本、滑点、涨跌停、停牌、调仓周期。
- 选股器结果必须能一键回测，而不是只展示命中。

### 第二阶段：把“盈利闭环”补起来

- 新增模拟组合/交易计划模块。
- 候选股输出不再只给方向，而是输出计划：触发价、失效价、目标区间、仓位建议上限、风险点。
- 新增复盘记录和错误标签。
- 策略每次修改必须对比修改前后的样本外表现。
- 方向卡增加历史准确率和同类样本统计。

### 第三阶段：补齐 A 股关键数据

- 财务报表：利润表、资产负债表、现金流、ROE、毛利率、负债率、经营现金流。
- 估值：PE、PB、PS、股息率、历史分位、行业分位。
- 行业/概念/板块：相对强度、板块资金、成分股联动。
- 市场宽度：涨跌家数、涨停跌停、均线多头比例、新高新低。
- 事件：公告分类、业绩预告、减持、回购、分红、解禁、龙虎榜。

### 第四阶段：强化体验与智能化

- 多图联动：个股、行业、指数、资金流同屏。
- 策略编辑器：参数、条件、因子、回测结果在一个页面闭环。
- 自然语言选股：自然语言 -> 条件表达式 -> 可解释 SQL/因子筛选。
- AI 解读只作为“总结器”，不能替代数据校验和回测。
- 一键日报：自选股变化、预警、候选股、风险事件、明日计划。

## 7. 建议的核心指标

产品是否能帮助盈利，应看这些指标：

- 数据指标：同步成功率、最新交易日覆盖率、跨源冲突率、异常修复时间。
- 策略指标：年化收益、最大回撤、夏普、胜率、盈亏比、换手率、单笔期望值。
- 稳定性指标：分年度收益、牛熊市收益、行业分布收益、样本外收益衰减。
- 风控指标：最大连续亏损、止损执行率、单票亏损上限、组合相关性。
- 用户行为指标：计划交易占比、复盘完成率、偏离计划次数、重复错误下降率。

## 8. 最关键的产品判断

当前项目已经有一个不错的“本地股票分析工作台骨架”，但距离顶级股票分析工具的核心差距不在页面数量，而在三件事：

1. 数据可信度：没有专业数据仓、质量校验、跨源对账和复权/公司行动体系。
2. 策略可信度：没有严肃回测、样本外验证、成本滑点、组合风控和策略归因。
3. 盈利闭环：没有交易计划、模拟组合、复盘纠错和策略迭代的强约束流程。

在不能接券商柜台的前提下，正确方向不是强行做交易，而是把系统打造成“本地投研 + 策略验证 + 预警 + 复盘”的闭环工具。先让每个结论都可追溯、每个策略都可验证、每次亏损都可归因，再谈 AI 预测和高级体验。
